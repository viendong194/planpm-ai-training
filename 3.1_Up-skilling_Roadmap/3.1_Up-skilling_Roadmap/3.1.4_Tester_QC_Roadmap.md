# 3.1.4. LỘ TRÌNH ĐÀO TẠO CHO TESTER / QC

Trong mô hình Agentic SDLC, Tester/QC chuyển từ "Người tìm bug ở giai đoạn cuối" thành **"Trust Validator — Bug Preventer (Shift-Left)"** — người gác cổng chất lượng từ Spec đến Production. Khi tốc độ phát triển tăng nhờ AI, vai trò QA càng quan trọng để đảm bảo tốc độ không đi kèm suy giảm chất lượng.

> **Phạm vi tài liệu này:** Tập trung vào **chuyển đổi tư duy + kỹ năng cốt lõi** cho role. Cách áp dụng tool / workflow / test pattern cụ thể nằm ở các chapter chuyên môn:

## 1. Chuyển đổi tư duy (Mental Model Shift)

3 shift quan trọng nhất khi Tester/QC chuyển sang Agentic SDLC:

### 1.1. Từ "Bug Hunter" → "Bug Preventer (Shift-Left)"

| Tư duy cũ | Tư duy mới |
|---|---|
| Vào dự án ở giai đoạn cuối, tìm bug | Vào dự án ngay từ Spec — review Spec testability |
| Đếm số bug phát hiện = giá trị | Đếm số bug **chặn ở Spec/Test design** = giá trị |
| Test khi có code | Test design **trước** khi code (Test-First) |
| QC = "người gác cổng cuối" | QC = "kiến trúc sư niềm tin" suốt vòng đời |

**Hệ quả thực tế (ROI shift-left):**
- 1 bug phát hiện ở Spec phase: chi phí 1×
- 1 bug phát hiện ở Code phase: chi phí 5-10×
- 1 bug phát hiện ở Production: chi phí 100-1000×

→ Đầu tư 4 giờ review Spec tiết kiệm **hàng tuần** debug production.

### 1.2. Từ "Manual Tester" → "Test Strategist + Quality Engineer"

- **Trước:** Click qua 50 test case manual mỗi sprint.
- **Sau:** Design test strategy (high-level), AI sinh test case + automation script, bạn **review correctness** + run + analyze.
- **Skill mới quan trọng nhất:** Verify AI test có đúng intent của Spec không (vì AI test cũng có thể hallucinate).

### 1.3. Từ "Trusting Tests" → "Testing the Tests"

> **Insight quan trọng:** AI sinh test cũng có thể bị hallucinate. Test pass không có nghĩa là code đúng — có thể test cũng sai.

> 💡 **Ví dụ minh họa concept:**
> Spec yêu cầu: *"Refund với amount > original → throws ValidationError"*. AI có thể sinh test:
> ```
> AI generated: expect(refund.status).toBe('FAILED')   ← AI giả định status, không match Spec
> Đúng phải là: expect(...).rejects.toThrow(ValidationError)  ← Match đúng Spec
> ```
> Test "pass" với code AI sinh sai → false sense of security. **QA phải review từng test** trước khi commit.

## 2. Kỹ năng cốt lõi cần phát triển

### 2.1. Adversarial Mindset — Tư duy "phá hoại"

- **Vì sao quan trọng:** AI sinh happy path tốt; nhưng edge case + adversarial scenario thường bị miss. QA phải **chủ động tìm cách break system**.
- **Cần phát triển:** Thinking như attacker (input attack, logic attack, business logic attack); chủ động prompt AI đóng vai adversary.

> 💡 **Ví dụ minh họa concept:** Cho feature subscription renewal, prompt AI: *"Bạn là Black-hat security tester + 1 angry user. Liệt kê 15 cách tấn công feature này (input/logic/business)"*. AI thường tìm ra ≥ 3 vector mà Spec không cover.

### 2.2. Edge Case Discovery — Domain-specific, không generic

- **Vì sao quan trọng:** Edge case generic ("null input") không bắt được edge case domain-specific. Mỗi domain có "blind spot" riêng (vd: payment có race condition refund, healthcare có HIPAA boundary).
- **Cần phát triển:** Maintain Edge Case Catalog per domain; review Catalog trước khi design test mới; contribute edge case mới phát hiện trở lại Catalog.

> 💡 **Ví dụ minh họa concept** (payment domain):
> - Refund sau cutoff period (> 30 ngày)
> - Double refund race condition (2 request < 100ms)
> - Partial refund tổng vượt quá original amount
>
> Đây là edge case người quen domain payment biết; AI generic không tự sinh được nếu Spec không gợi mở.

### 2.3. AI-Generated Test Review — Skill khác biệt nhất

> **CRITICAL skill** — đây là kỹ năng khác biệt nhất giữa QA truyền thống và QA Agentic.

- **Vì sao quan trọng:** AI test có thể hallucinate giống AI code. Test pass + code sai → false confidence cao nhất.
- **Cần phát triển checklist review** mỗi test do AI sinh:
  - Test name match Spec language?
  - Assertion match Spec exactly (status FAILED vs throw)?
  - Setup data hợp lệ với Spec?
  - Edge case có thật sự edge (không phải tự bịa)?
  - Mock realistic (không bịa response shape)?
  - Test isolated (no shared state, no order dependency)?
  - Cleanup proper?
  - Negative path đúng error message theo Spec?
  - Async/timing logic correct?
  - Coverage delta thực tế (không fake bằng test trivially passing)?

### 2.4. Coverage Interpretation — Số liệu nói gì thật sự

- **Vì sao quan trọng:** 80% coverage là **false confidence** nếu phần 80% là code dễ test, miss critical path.
- **Cần phát triển:**
  - Phân biệt line coverage / branch coverage / **critical path coverage** ([Glossary §9](../../Chapter_0_Introduction/Glossary_and_Definitions.md#9-critical-path--định-nghĩa--quy-trình-xác-định))
  - Hiểu mutation testing — **mutation score** mới là metric chất lượng thật của test (không phải coverage %)

> 💡 **Ví dụ minh họa concept:** Coverage 80% + mutation score 50% = test rất yếu (chỉ verify code chạy, không verify logic đúng). Coverage 70% + mutation score 85% = test mạnh hơn rõ rệt.

### 2.5. Synthetic Data Generation — PII-safe test data

- **Vì sao quan trọng:** Test cần data realistic để cover edge case, nhưng không được nạp PII vào AI/git.
- **Cần phát triển:** Phân loại data theo level L1-L4 ([Chapter 7.1](../../Chapter_7_Governance_and_Security/7.1_AI_Data_Security_and_Privacy_Rules.md)); biết khi nào synthetic / khi nào anonymize / khi nào absolutely no production data.

### 2.6. Anti-flaky Test Design

- **Vì sao quan trọng:** AI tạo test với tốc độ cao → flaky test risk cao. Flaky test bị "ignore" → mất confidence dần.
- **Cần phát triển:**
  - Tránh hard-coded timing (sleep/wait); dùng polling với timeout
  - Test isolation (transaction rollback hoặc DB-per-test)
  - Mock at boundary; integration test với retry policy
  - Seed random data với fixed value cho reproducibility

## 3. Anti-patterns thường gặp khi chuyển đổi

| # | Anti-pattern | Triệu chứng | Cách phòng |
|---|---|---|---|
| 1 | **Trust AI test correctness without review** | Test pass + code sai = false confidence | Bắt buộc review checklist 2.3 cho mọi test do AI sinh |
| 2 | **Coverage chasing** | 90% coverage nhưng miss critical path | Track Critical Path Coverage riêng; dùng mutation testing |
| 3 | **Generated nhưng không run** | AI sinh test, commit, không run → test theoretical | CI bắt buộc run mọi test mới merged |
| 4 | **Shared state trong test** | Flaky test, fail intermittent | Mỗi test isolated, transaction rollback |
| 5 | **Skip negative testing** | Chỉ test happy path, prod fail edge case | 1 happy : 2-3 negative test ratio per requirement |
| 6 | **Generic edge case** | Test "null input" cho mọi feature, không domain-specific | Maintain Edge Case Catalog per domain |
| 7 | **Không test data hygiene** | PII data trong test → leak risk | Synthetic data only; scan trước khi commit |
| 8 | **Mock bịa response** | Mock có shape sai docs → test pass, prod fail | Mock từ real API recording (Pact/WireMock) |
| 9 | **Test code không refactor** | Test code rối, không maintainable | Test code = production code; review quality, refactor đều |
| 10 | **Không track flaky test** | Flaky test bị "ignore" → mất confidence | Dashboard flaky rate; > 1% = action item |
| 11 | **AI sửa test cho code pass** | Code fail test → AI sửa test cho pass = bypass | Test files locked; cần Human approval để modify |
| 12 | **Không participate Spec review** | QA vào dự án ở Test phase, miss prevention chance | QA phải dự Spec Gate review |

## 4. Hành trình phát triển năng lực (L1 → L5)

> Chi tiết khung 5 level + assessment + badge: xem [Chapter 3.1](../3.1_Up-skilling_Roadmap_Details.md). Bảng dưới mô tả vai trò QA ở từng level.

| Level | Vai trò ở level này | Milestone |
|---|---|---|
| **L1 Apprentice** 🌱 | Học cách review test do AI sinh; chạy test có sẵn dưới giám sát | Hoàn thành review 1 test suite nhỏ với mentor pair |
| **L2 Practitioner** 🥉 | Tự design test strategy cho module nhỏ; biết verify AI-generated test | Solo deliver 1 test suite đạt Critical Path Coverage 100% |
| **L3 Proficient** 🥈 | Tự chủ test cho module phức tạp; áp dụng adversarial mindset; contribute Edge Case Catalog | Lead test cho 1 module lớn + ≥ 1 Edge Case Catalog entry |
| **L4 Expert** 🥇 | Define test standards cho team; coach junior; lead automation framework | Coach ≥ 2 QA lên L2; ≥ 5 Skills publish; lead test cho 1 project end-to-end |
| **L5 Master** 💎 | Define test strategy cấp SBU; thought leadership Quality Engineering | Lead transformation Quality practice ≥ 1 SBU; thought piece (talk/article) |

## 5. Gợi ý bài tập thực hành

> ⚠️ **Đây là gợi ý drill** để team tham khảo, không phải bài tập bắt buộc. Mỗi team có thể design drill phù hợp với context dự án và level học viên.

### Drill 1: Edge Case Exploration
**Đầu vào:** Feature spec 1 trang.
**Yêu cầu:** Dùng adversarial prompting sinh ≥ 20 edge case; map vào 3 category (Input/Logic/Business attack); flag edge case CHƯA có trong Spec gốc.

### Drill 2: Test Review (find AI hallucination)
**Đầu vào:** 1 test suite 30 test do AI sinh, có 4 lỗi ẩn (instructor cài: 2 assertion sai, 1 mock bịa, 1 missing edge case).
**Yêu cầu:** Review theo checklist 2.3 trong < 1 giờ. Phát hiện 4/4 lỗi với suggested fix.

### Drill 3: Synthetic Data Design
**Đầu vào:** Schema bảng có PII fields + bảng L2 internal.
**Yêu cầu:** Sinh synthetic dataset với phân loại đúng L1-L4; distribution match production stats ±10%; pass PII scan.

### Drill 4: Adversarial Prompt
**Đầu vào:** API spec endpoint (vd: payment refund).
**Yêu cầu:** Prompt AI tìm 5 cách break system; verify mỗi cách bằng test thực; ≥ 3/5 thực sự exploit được.

### Drill 5: Critical Path Mapping
**Đầu vào:** Spec module có 8 user stories.
**Yêu cầu:** Identify Critical Paths với rationale; mỗi path có justification thuyết phục Tech Lead; plan 100% coverage.

### Drill 6: Mutation Testing Setup
**Đầu vào:** 1 module có test suite 80% line coverage.
**Yêu cầu:** Setup mutation testing tool (Stryker/Pitest theo stack); analyze report; identify ≥ 5 surviving mutants → đề xuất test bổ sung.

## 7. Hướng phát triển sự nghiệp sau L5

| Hướng | Mô tả | Yêu cầu thêm |
|---|---|---|
| **Chief QA Architect / VP Quality** | Define test strategy + tooling toàn công ty | L5 ≥ 2 năm + ≥ 3 domain Edge Case Catalog đóng góp |
| **SDET (Software Dev Engineer in Test)** | Build internal test framework / automation tooling | Strong coding (≥ L4 Dev) + contribute Kari Inspector core |
| **Performance Engineer** | Specialize performance + scalability | Tools: k6/JMeter advanced, observability, system design |
| **Security Test Engineer** | Pentest, security automation | OSCP / CEH cert, tools: Burp, Metasploit |
| **Test Data / DataOps Engineer** | Synthetic data, test data lifecycle | Strong DB + privacy regulations expertise |
| **Site Reliability Engineer (SRE)** | Pivot sang Production reliability | Strong observability + on-call experience |

> [!IMPORTANT]
> Tester của tương lai không phải là người tìm ra nhiều lỗi nhất, mà là người **ngăn chặn nhiều lỗi nhất** ngay từ khi chúng còn nằm trên trang giấy đặc tả.

> [!TIP]
> Trong môi trường AI, skill khác biệt nhất của QA giỏi là khả năng **"test the tests"** — verify rằng AI test đo đúng intent của Spec. Bạn không chỉ là "QA của code", mà còn là "QA của AI test". Skill này không thể delegate cho AI.
