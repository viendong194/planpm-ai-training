# 3.1.3. LỘ TRÌNH ĐÀO TẠO CHO DEVELOPER (AI PILOT)

Trong mô hình Agentic SDLC, Developer chuyển từ "Người viết code" thành **"AI Pilot — Code Director + Reviewer"** — người thiết kế bộ khung, điều phối AI, và là người thẩm định cuối cùng cho mọi dòng mã. Giá trị đo bằng **độ tin cậy và hiệu năng** của hệ thống điều phối được, không phải số dòng code viết tay.

## 1. Chuyển đổi tư duy (Mental Model Shift)

3 shift quan trọng nhất khi Developer chuyển sang Agentic SDLC:

### 1.1. Từ "Code Author" → "Code Director + Reviewer"

| Tư duy cũ | Tư duy mới |
|---|---|
| Giá trị = số dòng code viết được | Giá trị = số bug bạn **chặn** được trong AI output |
| Pride trong việc tự gõ từng ký tự | Pride trong việc design Skill / Rule giúp toàn team sinh code đúng |
| Đo năng suất bằng commits/day | Đo năng suất bằng PR merged / Quality Gate pass |
| "Tôi viết code" | "Tôi điều phối AI viết code; tôi review & ký duyệt" |

**Cảnh báo phổ biến:** Cảm giác "không làm gì" trong những tuần đầu — đây là **review fatigue + impostor syndrome** điển hình. Đừng nhầm cảm giác này với "không có giá trị". Bạn đang chuyển từ contribution rõ ràng (code) sang contribution ẩn (judgment, review, direction) — giá trị **cao hơn** nhưng khó visualize.

### 1.2. Từ "Stack Overflow" → "Spec + Skills là source of truth"

- **Trước:** Gặp bug → Google → Stack Overflow → copy-paste → adapt.
- **Sau:** Gặp bug → check Spec (có cover case này không?) → check Skills hub (có pattern xử lý không?) → consult AI với context spec + project rules.
- **Hệ quả:** Stack Overflow vẫn dùng nhưng **chỉ để hiểu concept**, không copy code. Code luôn từ AI dưới context dự án.

### 1.3. Từ "Solo Hero" → "Pair với AI"

Mọi feature build với AI ngay từ đầu, không phải xong rồi mới hỏi AI sửa.

- **Cycle nhỏ:** Prompt → AI sinh → bạn review → modify → prompt tiếp. Mỗi cycle ≤ 30 phút.
- **Đối thoại liên tục:** Đừng "send 1 prompt khổng lồ rồi mong AI làm xong". Break nhỏ.
- **AI là Junior dev:** Đừng kỳ vọng AI hiểu hết context. Bạn nạp context, AI thực thi. Nếu AI sai 3 lần → 99% là context bạn cung cấp thiếu, không phải AI dở.

## 2. Kỹ năng cốt lõi cần phát triển

### 2.1. Test-First Mindset — Test viết trước, code viết sau

- **Vì sao quan trọng:** Test là **rail** giữ AI đúng hướng. Code AI sinh phải pass test → giảm hallucination drastically.
- **Cần phát triển:** Đọc và phân tích test trước khi prompt; viết hoặc enhance test cases theo Spec; resist thói quen "code trước, test sau".

> 💡 **Ví dụ minh họa concept** (không phải workflow bắt buộc):
> Khi nhận task implement RefundService, đọc test suite có sẵn → hiểu expected behavior → mới prompt AI sinh code pass test. KHÔNG prompt sinh code rồi viết test theo behavior code AI sinh.
> *Workflow chi tiết 6 bước: tham chiếu [Chapter 4.1](../../Chapter_4_Production_Workflows/4.1_Greenfield_Project_Workflow_6_Steps.md).*

### 2.2. Prompt Iteration Discipline — Quy tắc 3 lần thử

- **Vì sao quan trọng:** Khi AI sai, dev có xu hướng prompt thêm để patch. Càng patch càng lệch, code rối.
- **Cần phát triển:**
  - Lần 1 fail: prompt lại với context bổ sung
  - Lần 2 fail: phân tích — Spec mơ hồ? Context thiếu? Approach sai?
  - Lần 3 fail: **STOP** — không patch nữa. Lựa chọn: quay lại Spec, viết tay skeleton, hoặc reset clean state.
- **Tuyệt đối tránh:** "Patch" lần 4-5-6 — code sẽ nhồi accidentally, debug khổ về sau.

### 2.3. Architecture-aware Review — Nhìn xa hơn AI

- **Vì sao quan trọng:** AI chỉ thấy code trong context window, **không** thấy kiến trúc tổng. AI có thể sinh code đúng feature nhưng break boundary, tạo circular dependency, gây N+1 query.
- **Cần phát triển:** Khả năng review 4 lớp:
  - **Lớp 1** Syntax + Convention (AI Auto Review handle)
  - **Lớp 2** Logic Correctness — code đúng requirement Spec không, edge case handled
  - **Lớp 3** Architecture Awareness — layer boundary, dependency direction, module cohesion
  - **Lớp 4** System-level — N+1, cache, concurrency, observability

> 💡 **Ví dụ minh họa concept:** AI sinh code có thể implement đúng "controller gọi DB lấy order" — nhưng vi phạm Clean Architecture (controller không nên gọi DB trực tiếp, phải qua service layer). Lớp 3 review bắt được điều này.

### 2.4. Security-aware Review — Bắt điểm yếu AI

- **Vì sao quan trọng:** AI có những điểm yếu cụ thể về security mà Auto Review không bắt hết. Dev phải chủ động check.
- **Cần phát triển checklist quen thuộc cho mọi PR auth/payment/PII:**
  - Input validation (Zod/Joi/Pydantic schema)
  - IDOR (resource ownership check)
  - Race condition trong financial code (lock / optimistic locking)
  - Rate limit cho public endpoint
  - Secret hardcoded trong test
  - Sensitive data trong log
  - Error response không expose stack trace

> 💡 **Ví dụ minh họa concept:** AI sinh API endpoint refund kèm middleware auth → có vẻ secure. Nhưng manual check phát hiện endpoint không verify `refund.user_id == current_user.id` → IDOR vulnerability. Auto Review không bắt được vì syntax đúng.

### 2.5. Skill Contribution — Tích lũy tri thức team

- **Vì sao quan trọng:** Mỗi dev viết prompt riêng, không share → mỗi người có "ngôn ngữ AI" khác nhau, code style không đồng nhất.
- **Cần phát triển:** Đóng gói prompt + context + constraints hữu ích thành Skill share lên Kari Hub. Quy tắc: 1 Skill = 1 task narrow, versioned, test trong ≥ 2 dự án trước khi publish v1.0.

### 2.6. Pair Programming với AI — Effective patterns

- **Vì sao quan trọng:** Dùng AI hiệu quả ≠ prompt rồi accept output. Cần đối thoại structured.
- **Cần phát triển 3 pattern:**
  - **"Explain then Code"** — prompt AI giải thích approach + tradeoffs trước khi sinh code → catch hallucinated approach sớm
  - **"Diff Review"** — prompt AI generate diff + giải thích từng hunk → force AI articulate logic
  - **"Devil's Advocate"** — prompt AI tự critique (5 weakness) → second opinion

## 3. Anti-patterns thường gặp khi chuyển đổi

| # | Anti-pattern | Triệu chứng | Cách phòng |
|---|---|---|---|
| 1 | **"AI did it" mindset** | "Bug này AI viết, không phải tôi" | Bạn approve PR = bạn own code; AI là tool, không phải scapegoat |
| 2 | **Review chỉ syntax, không semantics** | PR pass review nhưng break business logic | Bắt buộc 4-layer review cho PR business-critical |
| 3 | **Long unbroken AI sessions** | Sau 10 prompt, AI drift, code không consistent | Break sau mỗi 30 phút; commit small, prompt small |
| 4 | **Không share Skills lên hub** | Mỗi dev tự viết prompt riêng → chất lượng không đồng nhất | KPI cá nhân: ≥ 1 Skill contribute / quarter |
| 5 | **Bandaid fix khi AI fail** | Patch 5-7 lần, code rối | Áp dụng 3-Attempt Rule; không patch lần 4 |
| 6 | **Skip Test-First** | Code không match Spec, fix lại tốn 2-3× thời gian | Test luôn viết trước; CI block PR không có test |
| 7 | **Trust AI cho security / concurrency** | SQL injection, race condition lọt production | Hand-verify mọi code chạm 9 điểm security |
| 8 | **Không version control prompts** | Prompt cũ work, prompt mới không, không biết tại sao | Commit prompt vào repo; treat như code |
| 9 | **AI giải bài bằng cách thay đổi test** | Test fail → AI sửa test cho pass thay vì fix code | Lock test files; AI không sửa test trừ khi human approve |
| 10 | **Không review AI's "thinking"** | Approve dựa vào code "trông được" | Áp dụng "Explain then Code" pattern |
| 11 | **Quên rebuild context khi switch task** | Cùng AI session → AI nhầm context giữa 2 feature | Mỗi feature = AI session mới; nạp lại Spec + Skills |
| 12 | **Disable Auto Review để merge nhanh** | Bypass quality, accumulate tech debt | Bypass = cảnh cáo; chỉ allow Emergency Bypass |

## 4. Hành trình phát triển năng lực (L1 → L5)

> Chi tiết khung 5 level + assessment + badge: xem [Chapter 3.1](../3.1_Up-skilling_Roadmap_Details.md). Bảng dưới mô tả vai trò AI Pilot ở từng level.

| Level | Vai trò ở level này | Milestone |
|---|---|---|
| **L1 Apprentice** 🌱 | Học prompt cơ bản, dùng AI sinh code dưới giám sát; chưa review độc lập | Hoàn thành 1 small task (≤ 100 LOC) với mentor pair |
| **L2 Practitioner** 🥉 | Tự deliver feature module nhỏ, biết Test-First + 3-Attempt Rule | Solo deliver 1 feature pass Code Gate (Quality ≥ 8.0) |
| **L3 Proficient** 🥈 | Tự chủ feature phức tạp; review Layer 2 cho peer; áp dụng 4-layer review thuần thục | Lead 1 module lớn + ≥ 1 Skill contribute lên Hub |
| **L4 Expert** 🥇 | Lead multi-module project; coach junior; define standards/Skills cho team | Coach ≥ 2 dev lên L2; ≥ 5 Skills publish; lead 1 project end-to-end |
| **L5 Master** 💎 | Set technical direction cấp công ty; contribute Kari ecosystem; thought leadership | Lead transformation ≥ 1 SBU; contribute Kari core OR thought piece |

## 5. Gợi ý bài tập thực hành

> ⚠️ **Đây là gợi ý drill** để team tham khảo, không phải bài tập bắt buộc. Mỗi team có thể design drill phù hợp với context dự án và level học viên.

### Drill 1: AI Code Audit
**Đầu vào:** 1 file 100-150 LOC do AI sinh, có 3 bugs ẩn (logic / security / performance).
**Yêu cầu:** Find cả 3 trong < 30 phút bằng manual review (không dùng AI hỗ trợ).

### Drill 2: Prompt Iteration
**Đầu vào:** 1 bad prompt vague.
**Yêu cầu:** Cải thiện qua 3 phiên bản (thêm context → constraint → output format + self-validation). v3 sinh code pass test + 0 critical security + handle 5 edge case.

### Drill 3: Skill Writing
**Đầu vào:** Yêu cầu — đóng gói 1 Skill cho task phổ biến (vd: RestController với pagination).
**Yêu cầu:** Document đầy đủ pre-conditions; test trong 2 endpoint khác nhau; 2 dev khác apply thành công không cần hỏi thêm.

### Drill 4: Bug Debugging với Hallucination
**Đầu vào:** 1 production bug — AI hallucinate gọi method/API không tồn tại.
**Yêu cầu:** Phân tích root cause; fix + add regression test; document lesson learned.

### Drill 5: Architecture Review
**Đầu vào:** 1 AI-generated PR 500 LOC.
**Yêu cầu:** Review 4-layer; produce review document với findings actionable mỗi layer.

### Drill 6: Security Pattern Audit
**Đầu vào:** 1 PR liên quan auth + payment.
**Yêu cầu:** Manual check 9 điểm security; produce checklist với pass/fail từng điểm.
